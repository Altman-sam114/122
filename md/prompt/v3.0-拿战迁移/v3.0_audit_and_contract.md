## 0. 本轮目标与非目标

目标：

- 建立从 `WWIIHexV0` 迁移到拿破仑战争题材的工程合同。
- 扫描当前运行时和主文档中的二战语义、二元阵营假设、默认阿登数据入口和玩家可见文案。

非目标：

- 不扩展 `Faction`、不改 `GamePhase`、不修改 `RuleEngine`、不新增滑铁卢 JSON。
- 不迁移 UI 视觉、不改 SpriteKit 渲染、不改 MapEditor schema。

## 2. 保留的工程权威

后续迁移必须保留这些运行时边界：

- `HexTile.controller` 和 `Division.coord` 仍是战术层权威。
- Region 只做战略聚合，不能替代 hex 移动、攻击、占领、视野或补给落点。
- `regionToTheater` 只作初始/基础战区映射，不表示运行时推进。
- `hexToTheater` 是运行时动态战区权威。
- `hexToFrontZone` 是部署层动态归属权威。
- 玩家、AI、MockAI 和未来聊天命令必须落到 `Command` / `ZoneDirective`，经 `WarCommandExecutor`、`CommandValidator`、`RuleEngine` 执行。
- Legacy Agent D 保留作回归参考，默认战争 AI 主路径不得退回旧管线。
- 不恢复 `organization`；拿战士气/疲劳只能在后续版本以新字段或可解释修正加入。

## 3. 硬编码审计清单

### 3.1 阵营与敌我关系

当前事实：

- `WWIIHexV0/Core/Faction.swift` 只有 `.germany` / `.allies`，`displayName` 返回 `Germany` / `Allies`。
- `Faction.opponent` 是二元反方 helper。
- `.opponent` 当前出现在 `SupplyRules`、`RegionSupplyRules`、`FrontLineManager`、`AgentContexts`、`ZoneCommanderAgent`、`WarCommandExecutor` 等路径。
- `DiplomacyState` 已存在国家/集团/关系结构，但其 `CountryProfile.faction` 和 `DiplomaticBloc.faction` 仍依赖二元 `Faction`；初始国家仍是 German Reich / United States / United Kingdom / Belgium。

风险：

- 多方滑铁卢至少需要 France / Anglo-Allied / Prussia / Neutral；二元 `opponent` 无法表达英荷联军与普鲁士同盟、临时中立、共同敌人和多敌对集团。
- 只扩展 enum 而不改敌我 helper，会让补给、撤退、前线和 AI 摘要继续把“非自己”错误折叠为单一敌人。

v3.1 合同：

- 新代码不得再直接调用 `faction.opponent` 判断敌我。
- 先建立统一关系 helper，例如 `DiplomacyState.isHostile(_:_:)`、`DiplomacyState.allies(of:)`、`DiplomacyState.enemies(of:)`，旧二元数据通过兼容桥返回原行为。
- `Faction.opponent` 可暂时保留给旧测试和旧数据，但应标注 legacy，并逐步从运行时主路径移出。
- 中立必须是独立状态，不能 fallback 到 `.allies`。

### 3.2 回合阶段与控制权

当前事实：

- `GamePhase` 仍是 `.germanAI` / `.alliedPlayer` / `.resolution`。
- `CommandValidator.phaseAllowsCommands` 写死 `.germanAI -> activeFaction == .germany`、`.alliedPlayer -> activeFaction == .allies`。
- `CommandExecutor.executeEndTurn` 写死 germany -> allies、allies -> germany 并更新对应 phase。
- `DataLoader.initialActiveFaction` 根据 `initialPhase` 在 `playerFaction` / `aiFaction` 之间二选一。
- `AppContainer` 的玩家可行动、玩家宏观命令、AI 自动运行都依赖 `.alliedPlayer` / `.germanAI`。
- `TurnManager.isAITurn` 同样按 germany/allies phase 判断。

风险：

- v3.1 若只新增 France/Prussia enum case，当前 phase 仍会拒绝这些势力行动。
- 玩家选择阵营和 AI 控制选项无法表达三方轮转。

v3.1 合同：

- 引入通用阶段语义，例如 `playerCommand`、`aiCommand`、`resolution`，或建立 `TurnController` / `FactionController` 桥接层。
- 旧 raw value 可保留解码兼容；UI 和新数据不得继续显示 German AI / Allied Player。
- `CommandValidator` 应只判断当前 phase 是否允许当前 active power 下命令，不直接写死具体阵营。
- `CommandExecutor` 的回合推进必须从 scenario turn order 或控制器队列读取下一势力。

### 3.3 单位、兵种、队形和战斗语义

当前事实：

- 运行时单位类型仍叫 `Division`。
- `ComponentType` 为 `.tank`、`.motorizedInfantry`、`.infantry`、`.artillery`。
- `Division.panzer`、`Division.motorized`、`Division.infantry`、`Division.artillery` helper 仍按二战构成生成单位。
- `Division.isArmor`、`isMechanizedHeavy`、`ZoneCommanderAgent.isMobile`、`UnitNode` 符号等仍依赖 tank / motorized。

风险：

- 拿战玩法需要 line infantry / light infantry / cavalry / artillery / guard / engineer / supply train；继续复用 tank/motorized 会让规则、AI 和 UI 都变成换皮。
- 一轮内把 `Division` 全局重命名风险过大，容易破坏 Codable、测试和 UI。

v3.3 合同：

- `Division` 作为源码兼容名短期保留，玩家可见显示改为 formation / corps / brigade / battery。
- 新增或迁移 `ComponentType` 时必须先设计旧模板兼容映射。
- 不恢复 organization；士气、疲劳、队形、弹药优先作为独立字段或战斗修正。
- 炮兵、骑兵、方阵、线列/纵队必须继续经 `Command` / `ZoneDirective` / `RuleEngine`，不能由 UI 或 AI 直接改战术状态。

### 3.4 经济、生产和后勤

当前事实：

- `EconomyResources` 字段为 manpower / industry / supplies。
- `ProductionKind` 仍包含 `infantryDivision`、`panzerDivision`、`motorizedDivision`、`artilleryDivision`、`supplyStockpile`。
- `EconomyPanelView` 玩家可见文本为 Economy、Manpower、Industry、Income IC、Production、Panzer/Motorized 对应按钮。

风险：

- 滑铁卢首发短战役更适合 reserve / reinforcement / ammunition / fatigue / forage，而不是现代工业生产。
- 若 v3.2 新增数据但 UI 仍显示 Industry / Panzer，会造成发布前残留。

v3.5 合同：

- 保留底层经济账本兼容时，必须提供拿战显示名和资源映射。
- 短战役优先做增援和预备队，生产队列可弱化或隐藏。
- 弹药、疲劳、马匹、补给车等只能通过规则层结算，不由 UI 直接扣账。

### 3.5 默认数据、DataLoader 和 MapEditor

当前事实：

- `DataLoader.loadInitialGameState()` 默认加载 `ardennes_v0_scenario` + `ardennes_v02_regions`。
- `loadArdennesDataSet()`、`loadScenarioDefinition()`、`loadArdennesV02Regions()` 均指向阿登资源。
- `DataLoader.validate(_ dataSet:)` 强校验 German / Allied supply source，并在 `scenario.id == "ardennes_v0"` 时校验 `guderian`。
- `DataLoader.fallbackComponents` 识别 `tank_division`、`panzer_division`、`motorized_division` 等二战模板。
- `RegionDataSet.toRegions()` 当前把 `owner == nil`、`controller == nil` fallback 到 `.allies`，这与中立语义冲突。
- `MapEditorGameResourceBridge` 默认资源是阿登；`MapEditorExporter` 默认 `alliedPlayer`、`allies`、`germany`；编辑器 UI 仍有摩托化师和 faction 二选一。

风险：

- v3.2 如果只新增 `waterloo_1815_*.json` 而不改默认入口，应用仍打开阿登。
- 中立 region fallback 到 `.allies` 会破坏 Neutral、普军迟到、据点初始争夺等拿战剧本语义。
- DataLoader 的阿登专用 validation 若复用于滑铁卢，会误报 supply/agent。

v3.2 合同：

- 新增拿战数据时不要覆盖阿登资源；先建立资源选择或 scenario manifest。
- `RegionDataSet.toRegions()` 必须先解决 neutral，或明确以新 schema 隔离 neutral 字段。
- 新 JSON id 使用 ASCII，例如 `waterloo_1815`、`power_france`、`region_hougoumont`、`commander_napoleon`。
- Project/resource 引用只能由唯一指定 Agent 修改，避免 `project.pbxproj` 冲突。

### 3.6 UI 和 SpriteKit 玩家可见残留

当前事实：

- `HUDView` 显示 `Ardennes V0` 和 Industry。
- `BoardScene` 标题是 `Ardennes V0 Board`。
- `RootGameView` accessibility label 是 `Ardennes V0 hex board`。
- `EconomyPanelView` 显示 Industry、Income IC、Production 和现代生产项。
- `UnitInspectorView`、`UnitTooltipView`、`UnitNode` 仍根据 tank / motorized 识别单位类别。
- `HexNode` supply source 标签仍是 `SUP A` / `SUP G`。

风险：

- 发布前“无主要二战文案残留”不只包括 JSON，也包括 accessibility label、日志、按钮、单位类型和地图标题。

v3.6 合同：

- UI 先引入显示名/主题 token 层，不在 View 中散落题材硬编码。
- SpriteKit 图层只读状态；点击和命令仍走 `AppContainer` / `RuleEngine`。
- 视觉验证需要 app 启动或截图，但当前规范禁止默认执行，必须作为未验证风险记录。

### 3.7 AI / 指令链

当前事实：

- 旧 Agent D 管线保留；v3.4 当前默认战争 AI 主路径是 `RulerAgent / StrategicPostureEnvelope -> StrategicPostureDecoder -> MarshalAgent / TheaterDirective -> decoder -> compiler -> ZoneDirective -> WarCommandExecutor -> RuleEngine`。
- `AgentContexts`、元帅摘要、补给、前线、部署和战区压力已开始通过 `DiplomacyState.isHostile/isFriendly` 判断敌我；`Faction.opponent` 只保留 legacy helper。
- `ZoneCommanderAgent`、`RulerAgent` 已有 France / Anglo-Allied / Prussia 等拿战 fallback 文案和 commander config，但完整拿战 personality 仍待后续。
- `WarCommandExecutor` 主体仍是唯一执行翻译器，不直接跳过规则系统。

风险：

- 三方或联军 AI 需要从 relation helper 取敌军和盟军，否则普军与英荷联军不能正确协同。
- Agent prompt / raw JSON / WarDirectiveRecord 中的战术名和 commander 名需要拿战化，但不能改成直接执行底层状态修改。

v3.4 合同：

- 所有上游 Agent 输出必须是 Codable envelope。
- decoder 失败只能 fallback，不执行半成品。
- 皇帝/总司令/参谋/军团长输出只塑形 directive，不直接改 `GameState`。
- `WarDirectiveRecord` 和 `AgentDecisionRecord` 继续作为审计记录。

## 5. v3.1 公共接口合同

v3.1 是高风险基础版本，建议先做接口合同再改源码：

1. 阵营/势力：
   - 若保守扩展 `Faction`，至少支持 `france`、`angloAllied`、`prussia`、`neutral`。
   - 若转向数据驱动 `PowerId`，必须提供旧 `Faction` 兼容桥，避免一轮内改穿全项目。
2. 敌我判断：
   - 建立 `DiplomacyState` 或 `CoalitionState` 的统一查询 API。
   - 新增代码只能调用关系 API，不直接用 `.opponent`。
3. 回合控制：
   - 从二元 phase 切到 active power + controller kind + turn order。
   - `CommandValidator` 只关心当前 active power 是否允许下命令。
4. 中立：
   - `HexTile.controller` 可继续为 nil 表示中立。
   - Region owner/controller 不得在 nil 时 fallback 到 `.allies`。
5. 数据：
   - scenario 必须能表达多个 faction/power 和 turn order。
   - 旧阿登数据通过 legacy path 加载，不污染拿战默认 path。

## 6. 并发分工边界

后续多 Agent 并发时建议按文件边界拆分：

- Rules Agent：`Core/Faction.swift`、`Core/GamePhase.swift`、`Rules/CommandValidator.swift`、`Rules/CommandExecutor.swift`、`Rules/SupplyRules.swift`、`Rules/FrontLineManager.swift`。不碰 UI。
- Data Agent：`Data/ScenarioDefinition.swift`、`Data/DataLoader.swift`、`Data/RegionDataSet.swift`、新增拿战 JSON。只读规则层。
- AI Agent：`Agents/`、`Turn/`。只通过 relation helper 读敌我，不直接写状态。
- UI / SpriteKit Agent：`UI/`、`SpriteKit/`。只读状态并调用 AppContainer 动作，不写规则。
- MapEditor Agent：`MapEditor/`。只遵守 Data schema，不发明另一套 schema。
- Project / Assets Agent：唯一修改 `WWIIHexV0.xcodeproj/project.pbxproj` 和 asset catalog 的角色。

公共规则：

- 多个 Agent 不得同时修改 `Faction`、`GamePhase`、`ScenarioDefinition` 或 `project.pbxproj`。
- 新增 Codable schema 必须记录旧数据兼容策略。
- 未做冲突检查前，不得声称可合并。

1. 先写 `Faction` / `PowerId` / `DiplomacyState` / `GamePhase` 的最小迁移设计。

5. 只做 `md/test/test.md` 允许的轻量检查；运行时正确性留给云端或人工授权重测试。
